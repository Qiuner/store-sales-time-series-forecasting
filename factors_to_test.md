# 需要预测与验证的因子

这个文档用于记录 `Store Sales - Time Series Forecasting` 当前已经确认会影响 leaderboard 分数的因子，以及下一步应优先测试的候选因子。

## 已验证因子

### 1. 最终融合权重

- 对象: `LightGBM / XGBoost` 最终加权比例
- 证据:
  - `version6`: `0.38022`
  - `version6.1` (`0.60 / 0.40`): `0.38041`
  - `version6.2` (`0.45 / 0.55`): `0.38017`
  - `version6.3` (`0.40 / 0.60`): `0.38015`
  - `version6.4` (`0.35 / 0.65`): `0.38016`
  - `version6.5` (`0.42 / 0.58`): `0.38016`
- 结论: 最终融合权重会稳定影响分数，当前最佳区间在 `0.40 / 0.60` 附近

### 2. 是否保留 `lag 730`

- 对象: 超长周期滞后特征 `730`
- 证据:
  - `version6.3`（保留 `730`）: `0.38015`
  - `version7.1`（去掉 `730`）: `0.38138`
- 结论: `lag 730` 提供了有效的长周期信息，移除后分数明显变差，应保留

### 3. 是否保留 `lag 365`

- 对象: 年周期滞后特征 `365`
- 证据:
  - `version6.3`（保留 `365`）: `0.38015`
  - `version7.2`（去掉 `365`）: `0.38074`
- 结论: `lag 365` 同样提供了有效的一年周期信息，移除后分数明显变差，应保留

### 4. 节假日特征是否存在噪声

- 对象: `selected_holidays` 的节假日集合大小与筛选方式
- 证据:
  - `version6.3`（保留当前 7 个候选节日）: `0.38015`
  - `version7.3`（只保留 4 个核心节日）: `0.38005`
- 结论: 当前节假日集合中存在噪声特征，缩减为核心节日后可以稳定提分

### 5. 是否保留 `transactions`

- 对象: 过去协变量 `transactions`
- 证据:
  - `version7.3`（保留 `transactions`）: `0.38005`
  - `version7.4`（移除 `transactions`）: `0.38110`
- 结论: 去掉 `transactions` 后分数明显变差，说明交易量序列当前仍然提供有效信息，应继续保留

## 下一步优先测试的候选因子

### 1. 节假日特征进一步筛选

- 原因: `version7.3` 已证明“减少节假日噪声”有效，但 4 个核心节日是否已经最优还未验证
- 建议实验:
  - 继续从 `version7.3` 出发，逐个移除 `nat_navidad / nat_dia trabajo / nat_futbol / nat_dia difuntos`
  - 或只测试 `3` 个核心节日组合
- 目标: 找到最优节假日最小子集

### 2. 其他过去协变量组合

- 对象: `oil`、`onpromotion` 及其不同组合方式
- 原因: `transactions` 已被验证应保留，但其他协变量的边际贡献仍未完全拆清
- 建议实验:
  - 继续做单因子消融或组合消融
  - 保持核心节日、长周期 `lags` 和融合权重不变
- 目标: 进一步识别哪些外部时间序列特征真正有贡献

### 3. 模型内 `lags` 组合

- 对象: 各组 `LightGBMModel` / `XGBModel` 的 `lags` 配置
- 原因: 当前是多组 `lags` 并行集成，但每组的边际贡献还不清楚
- 建议实验:
  - 删掉一组短周期 `lags`
  - 删掉一组长周期 `lags`
  - 对比是否有冗余分支
- 目标: 判断当前重型集成里是否存在可以裁剪的分支

### 4. `LightGBM` 与 `XGBoost` 参数

- 原因: 目前已经确认“模型间权重”重要，但“模型内参数”还没系统测
- 建议实验:
  - 优先微调 `learning_rate`、`max_depth`、`n_estimators`
  - 一次只改一个方向，避免多因子混杂
- 目标: 判断提升空间来自融合，还是也来自单模型质量

## 当前实验原则

- 所有新版本尽量基于 `version6.3`
- 每次只改一个因子
- 每次保留提交用 `csv`
- 记录版本号、改动点、Kaggle public score、结论

## 当前推荐顺序

1. 先细化节假日最小子集
2. 再看其他协变量组合
3. 再看是否有必要裁剪其他 `lags` 分支
4. 最后再考虑模型参数微调
